# Task Init (Session-Start Refresh)

This skill refreshes project context from the issue tracker at the start of a session. It ensures the agent has an accurate view of board state before doing any work.

## When to Run

**Always** — at the start of every new conversation/session, as part of the Initial Actions contract in `holicode.md` (step 2).

This is not user-invoked. It runs as part of the standard session boot sequence. Agents should execute this skill after detecting a `.holicode/` project, before engaging with the user's request.

## When NOT to Run

- Mid-session (context is already loaded and being maintained)
- If `.holicode/` directory does not exist (not a HoliCode project)

## Zone Model

activeContext.md and progress.md use two zone types (marked by HTML comments):

- **Generated zones** (`<!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->`): Regenerated deterministically from tracker state. task-init OVERWRITES these sections entirely on every run.
- **Append-only zones** (`<!-- APPEND-ONLY — ... -->`): task-init NEVER modifies these sections. Workflows append timestamped entries here.

This separation ensures parallel worktree merges are conflict-free.

## Steps

### 1. Load state files

Read the three session-start state files:
- `.holicode/state/activeContext.md` — current focus, recent changes, next steps
- `.holicode/state/progress.md` — milestones, completion status
- `.holicode/state/WORK_SPEC.md` — board snapshot, tracker issue cache

### 2. Refresh from board (if tracker available)

Read `.holicode/state/techContext.md` to determine the active `issue_tracker` provider.

If an external tracker is configured and available (`vibe_kanban`, `github`):

1. **Fetch current board**: Use the provider's list operation (`list_issues` or equivalent) to get all issues with current statuses, relationships, and blockers.

2. **Regenerate Generated zones in activeContext.md**: Overwrite ONLY the sections marked `<!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->`. Do NOT read previous content of these sections — regenerate from tracker state:

   **`## Current Focus`** — Issues in "In progress" status:
   ```
   ## Current Focus
   <!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->
   - HOL-37: Session-Start State Refresh [In progress]
   - HOL-16: Data Ingestion Skill [In progress]
   ```

   **`## Ready to Start`** — Issues in "To do" that are unblocked:
   ```
   ## Ready to Start
   <!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->
   - HOL-19: Rapid-Prototype Lane [To do, unblocked]
   ```

   **`## Blockers`** — Blocking relationships:
   ```
   ## Blockers
   <!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->
   - HOL-39 blocked by HOL-38
   ```
   Or `- None` if no blockers exist.

3. **Regenerate Generated zone in progress.md**: Overwrite ONLY the `## Current Status` section (marked `<!-- GENERATED BY task-init -->`). Derive from board state:

   ```
   ## Current Status
   <!-- GENERATED BY task-init — DO NOT EDIT MANUALLY -->
   - **Phase**: [inferred from active epics/milestones]
   - **Version**: [from mb_meta or techContext]
   - **Status**: [On Track / At Risk — based on blocker count and in-progress ratio]
   ```

   Do NOT modify Append-only zones in progress.md (`Completed Milestones`, `Current Milestones — IN PROGRESS`, `Known Issues & Blockers`).

4. **Do NOT modify Append-only zones**: Leave `## Recent Changes`, `## Immediate Next Steps`, `## Open Questions/Decisions` in activeContext.md untouched. Leave append-only zones in progress.md untouched.

5. **Refresh WORK_SPEC.md**: Invoke the `issue-sync` skill to update `.holicode/state/WORK_SPEC.md` with the current board snapshot.

If `issue_tracker: local` or tracker is unavailable, skip this step — the loaded state files are the best available context.

### 3. Board triage (derive status + recommend next)

Using the fetched board data, perform lightweight triage:

1. **Derive epic status from children**: For each epic, check sub-issue statuses.
   - All children Done → epic should be Done
   - Any child In progress → epic should be In progress
   - All children To do → epic stays To do
   - Flag any mismatches (epic says "To do" but has children "In progress")

2. **Fix stale statuses**: If an issue is "In review" but its PR was merged (check `awareness-inbox` or session context), recommend advancing it to QA (not directly to Done). QA means merged but awaiting validation. Done requires explicit QA sign-off. See `holicode.md` → "Issue Lifecycle & Status Flow" for the full status model.

3. **Identify unblocked ready items**: Issues in "To do" where:
   - No `blocking` relationship from an incomplete issue
   - Parent epic is not blocked
   - Sorted by priority (urgent > high > medium > low)

4. **Recommend dispatch batch**: Based on unblocked items, suggest 1-3 issues that could be dispatched in parallel (independent, no file overlap). Include:
   - Issue reference and title
   - Suggested executor (default: CLAUDE_CODE)
   - Whether any have dependency relationships requiring sequencing

This triage is **advisory only** — the agent presents recommendations, the human (or orchestrating session) decides.

### 4. Present orientation summary

Output a brief summary so the agent (and user) can orient:

```
Session Start:
- Epics: [count active] ([with completion ratios, e.g., "HOL-11: 2/4 done"])
- In progress: [count] ([brief titles])
- Recently completed: [count]
- Ready to start: [count] ([brief titles, priority-sorted])
- Blockers: [any blocking relationships or "none"]
- Recommended next: [1-3 items with rationale]
```

### 5. TPM health check (optional)

After presenting the orientation summary, check if a TPM report should run:

1. Read `.holicode/state/delegationContext.md` and find the `### Autonomous Roles` > `#### TPM` section
2. If TPM `Enabled: true` AND `Cadence: session_start`:
   - Invoke the `tpm-report` skill
   - Append a 2-3 line health summary to the orientation output (e.g., "TPM Health: GREEN — no critical findings")
3. If TPM is not enabled, or cadence is `on_demand`, skip silently

This step adds < 5 seconds when skipped, and < 30 seconds when the TPM report runs.

### 6. Check handoffs

Look in `.holicode/handoff/active/` for any pending handoff files. If found, mention them in the summary.

## Output

The skill does not produce a file artifact. Its output is:
1. **Side effects**: Regenerated Generated zones in `activeContext.md` and `progress.md`, and refreshed `WORK_SPEC.md` (if tracker was available). Append-only zones left untouched.
2. **Summary**: Orientation text with board triage recommendations presented to the agent/user

## Relationship to Other Skills

- **Delegates to**: `issue-sync` (for WORK_SPEC refresh), `tpm-report` (optional, if cadence == session_start)
- **Uses**: `issue-tracker` provider (for board read — read-only, no mutations)
- **Complements**: `intake-triage` (runs after task-init; task-init provides context, intake-triage classifies the user's request)
- **Feeds into**: `workspace-orchestrate` (dispatch recommendations from triage can be executed via workspace-orchestrate)

## Constraints

- Read-only against the tracker — never mutate issues during init (status fixes are recommendations, not auto-applied)
- Keep it fast — target < 30 seconds wall time for the refresh
- If tracker is unavailable, degrade gracefully to loaded state files
- Do not ask the user questions — this is a silent boot step
- NEVER modify append-only zones — only regenerate generated zones
- Board triage recommendations are advisory — dispatch requires human or orchestrator confirmation
